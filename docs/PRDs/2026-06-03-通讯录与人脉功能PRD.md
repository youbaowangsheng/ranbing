## 一、产品概述

### 1.1 背景与目标

燃烧吧小程序已完成基础社交功能（好友关系、私信、社群），但缺乏系统化的联系人管理与人脉增值能力。本期功能旨在将现有 Connection 关系体系升级为完整的**通讯录 + 人脉网络**产品，核心目标：

- **降低社交门槛**：让用户清楚知道自己认识谁、如何触达
- **提升关系价值**：通过标签、分组、名片让联系人资产可运营
- **促进合作转化**：基于人脉统计驱动更多供需合作

### 1.2 现有资产盘点

| 资产 | 现状 | 可复用程度 |
|------|------|-----------|
| `User` 模型 | phone/wx_openid/nickname | 直接复用 |
| `Profile` 模型 | 一对一：real_name/company/position/avatar/教育背景/认证等级 | 直接复用，作为主名片 |
| `Connection` 模型 | user_a/user_b/conn_type/relation_strength/interact_count | 直接复用，作为好友关系 |
| `ProfileTag` | profile × tag 多对多（目前 tag_type 只区分供给/需求标签） | 扩展 tag_type=3 用于联系人标签 |
| `PrivateMessage` | 真实私信系统 | 直接复用 |
| `CommunityMember` | 社群成员角色体系 | 复用成员数据到人脉统计 |

## 二、功能模块

### 模块 1：通讯录（联系人管理）

#### 1.1 联系人列表

**功能描述：**
展示当前用户所有有效好友关系，按联系强度（relation_strength）和最近互动（last_interact_at）排序。

**数据来源：**
- 现有 `Connection` 表，`status=1`，按 `relation_strength DESC, last_interact_at DESC` 排序
- 关联 `Profile` 展示头像/公司/职位

**交互规则：**
- 列表展示：头像 + 名字 + 公司 + 职位 + 标签（最多显示3个）
- 点击进入联系人详情页
- 支持按"最近联系" / "按公司" / "按行业" 排序
- 搜索：按名字/公司/职位全文搜索（前端本地过滤或调搜索接口）

**边界状态：**
- 空列表：引导文案"还没有好友，去供需页或活动里认识新朋友吧"
- 加载中：骨架屏

---

#### 1.2 好友申请管理

**功能描述：**
管理发出的好友申请和收到的申请，处理通过/拒绝。

**现状：**
现有 Connection 只有 `status=1(有效)/2(已删除)`，没有申请-待确认流程。

**新增数据流：**
- Connection.status 扩展：
  - `0 = 待确认`（发起方发起，等待对方确认）
  - `1 = 有效`（双向确认）
  - `2 = 已删除`
  - `3 = 已拒绝`（对方拒绝）
  - `4 = 已过期`（超时未确认）

**API 设计：**
```
POST   /api/v1/connections/          新增好友申请（from_user, to_user, conn_type）
GET    /api/v1/connections/pending/ 获取待确认的好友申请（收到的）
GET    /api/v1/connections/sent/    获取发出的申请（待确认/已拒绝）
POST   /api/v1/connections/{uuid}/accept/   接受申请 → status=1，is_mutual=True
POST   /api/v1/connections/{uuid}/reject/   拒绝申请 → status=3
DELETE /api/v1/connections/{uuid}/    删除关系 → status=2
```

**前端页面：**
- "通讯录" tab 新增"好友申请"入口（红点显示未读数量）
- 收到申请列表：申请人头像/名字 + 申请说明 + [接受] [拒绝]
- 发出申请列表：对方头像/名字 + 状态（待确认/已拒绝）

---

#### 1.3 联系人详情页

**功能描述：**
查看联系人完整档案，发起私信或社群互动。

**复用现有 `profile-view` 页面**，扩展：
- 新增"添加标签"按钮
- 新增"设置备注"按钮
- 新增"人脉关系路径"展示（如果有中间人，显示"你 → 张三 → 李四 认识王五"）

### 模块 2：联系人打标签与分组

#### 2.1 联系人标签系统

**功能描述：**
用户可对自己的联系人打自定义标签（如"投资人"、"技术合作"、"VC"），用于分类筛选和管理。

**数据模型复用：**
扩展现有 `ProfileTag` 的 `tag_type` 字段：
- `tag_type=1`：供给标签（已有）
- `tag_type=2`：需求标签（已有）
- `tag_type=3`：**联系人标签（新增）**

新增标签通过现有 `Tag` 表管理，无需新建表。

**标签数据结构：**
```python
ProfileTag(
    profile=当前用户.profile,
    tag=标签ID,
    tag_type=3,  # 联系人标签
    weight=1.0,
    is_ai_ext=False
)
```

**约束：**
- 每个联系人的同一标签只允许一条记录（unique_together 已有）
- 标签名称最大20字符，最多50个标签
- 单个联系人最多打10个标签

**API 设计：**
```
GET    /api/v1/profiles/tags/contacts/         获取当前用户的所有联系人标签列表
GET    /api/v1/profiles/contacts/{uuid}/tags/  获取某联系人的标签列表
POST   /api/v1/profiles/contacts/{uuid}/tags/ 添加标签 {tag_id: int}
DELETE /api/v1/profiles/contacts/{uuid}/tags/{tag_id}/ 删除标签
```

**前端交互：**
- 联系人详情页点击"标签"区域 → 弹出标签选择器（已选标签高亮 + 可搜索/新增）
- 通讯录列表每个联系人显示前3个标签
- 通讯录支持按标签筛选（类似活动分类筛选 UI）

### 模块 3：多种名片

#### 3.1 背景

现有 `Profile` 是每个用户一张主档案，**一对一关系**。本期新增"多名片"能力，让用户拥有多张不同场景使用的名片。

#### 3.2 数据模型

```python
class Card(models.Model):
    """用户多名片"""
    uuid = UUIDField(default=uuid4, unique=True)
    profile = ForeignKey(Profile, related_name='cards')
    name = CharField(max_length=64)              # 姓名（可与主档案不同）
    company = CharField(max_length=128, blank=True)
    position = CharField(max_length=128, blank=True)
    phone = CharField(max_length=32, blank=True)
    wechat = CharField(max_length=64, blank=True)
    email = CharField(max_length=128, blank=True)
    avatar = ImageField(upload_to='cards/avatars/', blank=True)
    bio = TextField(blank=True)                 # 一句话介绍
    tags = JSONField(default=list)              # 标签，如 ["创始人", "海归"]
    is_default = BooleanField(default=False)    # 是否默认名片
    visibility = SmallIntegerField(default=1)   # 1=公开 2=仅好友 3=仅自己
    created_at = DateTimeField(auto_now_add=True)
    updated_at = DateTimeField(auto_now=True=True)

    class Meta:
        ordering = ['-is_default', '-created_at']
```

**关键设计决策：**
- `Profile` 主档案作为"默认名片"（不可删除）
- `Card` 为额外名片，最多5张
- 每张名片有独立的 `visibility`（可见权限）
- 删除名片不影响 `Profile` 主档案

#### 3.3 API 设计

```
GET    /api/v1/cards/                  获取当前用户所有名片
POST   /api/v1/cards/                  创建名片
GET    /api/v1/cards/{uuid}/           获取某张名片详情
PUT    /api/v1/cards/{uuid}/            更新名片
DELETE /api/v1/cards/{uuid}/           删除名片（不能删除主档案对应的名片）
POST   /api/v1/cards/{uuid}/set_default/ 设置为默认名片
```

#### 3.4 前端页面

**"我的名片"页面：**
- 顶部名片展示：横向滑动（多张名片）
- 每张名片可点击编辑
- 底部"添加名片"按钮
- 每张名片可单独设置可见权限

**"分享名片"功能：**
- 生成分享链接/小程序码，他人无需登录即可查看（受 visibility 控制）
- 分享出去的页面为独立的 h5 页面（`/pages/card-share/card-share?uuid=xxx`）

### 模块 4：可见权限控制

#### 4.1 权限模型

在名片（Card）、供需（Supply）上增加 `visibility` 字段：

| 值 | 含义 | 可见范围 |
|----|------|---------|
| 1 | 公开 | 所有人 |
| 2 | 好友可见 | 双向 Connection 有效（is_mutual=True）|
| 3 | 仅自己 | 只有自己 |

**实现方式：**
在 DRF ViewSet 的 `get_queryset()` 里，根据当前用户身份动态过滤：
```python
def get_queryset(self):
    qs = Card.objects.filter(profile=self.request.user.profile)
    if self.action == 'list':
        # 非详情页，按 visibility 过滤
        if not self.request.user.is_authenticated:
            qs = qs.filter(visibility=1)
        else:
            # 获取当前用户好友UUID列表
            friends = get_friend_uuids(self.request.user)
            # 公开 + (好友可见 AND 是好友)
            qs = qs.filter(
                Q(visibility=1) |
                Q(visibility=2, profile__uuid__in=friends)
            )
    return qs
```

#### 4.2 用户控制层

**每张名片设置界面：**
- 可见范围选择器（单选）：公开 / 好友可见 / 仅自己
- 底部文字提示："好友可见：只有你们互为好友后才能看到这张名片"

**数据影响范围（Phase 2）：**
- Card 名片可见权限 ✅ 本期
- Supply 供需可见权限（已在需求文档中作为远期功能）
- Profile 主档案可见权限

### 模块 5：人脉统计

#### 5.1 统计指标

基于现有数据实时计算，不建新表（可缓存）：

| 指标 | 计算方式 | 展示 |
|------|---------|------|
| 好友总数 | Connection where status=1 and (user_a=user or user_b=user) count | 数字 + "位好友" |
| 有效人脉 | 同上 | 数字 |
| 本月新增 | Connection where created_at >= 本月1日 | "+N 本月新增" |
| 供需互动数 | Supply where profile=user.profile and match_count>0 count | "产生了 N 次匹配" |
| 社群参与 | CommunityMember where profile=user.profile count | "加入了 N 个社群" |
| 活动参与 | ActivityEnrollment where profile=user.profile count | "参加了 N 场活动" |
| 人脉圈层 | 按 relation_strength 分层：强关系(>0.8)/中关系(>0.5)/弱关系 | 环形图/条形图 |

#### 5.2 API 设计

```
GET /api/v1/profiles/network_stats/
```

返回：
```json
{
  "code": 0,
  "data": {
    "total_connections": 42,
    "new_this_month": 5,
    "communities_joined": 3,
    "activities_joined": 8,
    "supply_match_count": 12,
    "layers": {
      "strong": 12,
      "medium": 18,
      "weak": 12
    }
  }
}
```

#### 5.3 前端页面

**"人脉" Tab（新增）入口在"通讯录"页顶部：**
- 头部统计卡片：好友总数 + 本月新增（横排两个数字）
- 中部：人脉圈层分布图（三个横向条形：强关系/中关系/弱关系）
- 底部：活动参与数、社群加入数、供需匹配数

## 五、风险与依赖

| 风险 | 影响 | 缓解措施 |
